# Claude Codeで「案件管理システム」を1日で作ってみた（要件定義→Plan Mode→実装→公開まで）

これまで作ってきたのは、チュートリアル通りのサンプルアプリや、既存ツールの模写ばかりでした。今回は「自分で要件を決めて、実際に使えるものを1つ完成させる」ことを目標に、副業Webライター向けの案件管理Webアプリをゼロから作ってみました。

## 作ったもの

クラウドワークスやランサーズなど複数のクラウドソーシングサービスを使うフリーランス層向けの、案件管理Webアプリです。

- 案件の登録・一覧・詳細編集
- 応募状況管理（候補／応募中／面談／契約／不採用）
- 進捗管理（未着手／作業中／確認中／完了）
- 一覧のステータス・納期による絞り込み
- 納品日・納品先URLの記録
- 報酬・入金予定日・入金済みフラグの管理
- 月別売上の自動集計
- 進行中案件数・直近納期・未入金額・当月売上を表示するダッシュボード

公開URL: https://freelance-job-tracker-five.vercel.app
GitHub: https://github.com/shuu112475-afk/freelance-job-tracker

## 誰のためのツールか

クラウドワークスやランサーズなど複数のサービスを掛け持ちする副業・フリーランス層は、応募案件・返信状況・面談・納期・報酬・入金状況をそれぞれ別の場所で管理しがちです。その結果、応募先の重複、返信や納期の見落とし、未入金の放置が起こります。

このツールは、案件の発見から応募・選考・契約・作業・納品・入金・振り返りまでを1つの画面に集約し、「今、何を優先して確認すべきか」を迷わず把握できる状態を目指しました。

## 使ったClaude Codeの機能

### 要件定義書ベースの開発

いきなり実装させるのではなく、まず`requirements.md`という要件定義書を作成しました。背景・目的・スコープ・ユーザーストーリー・機能要件（優先度とフェーズ付き）・非機能要件・制約条件・画面遷移まで書き出してから開発に着手しています。以降は「requirements.md通りに実装して」の一言で、方向性がブレずに実装を進められました。

### CLAUDE.md（`/init`で生成）

`/init`コマンドでプロジェクト固有のCLAUDE.mdを生成し、以下のルールを追記しました。

- 要件定義書を要件の正（source of truth）とすること
- 実装フェーズはMVP → v1.0 → v2.0の優先順位に従うこと
- 複数ファイルにまたがる実装はPlan Modeで設計→承認→実装の順で進めること
- 1機能ずつ「実装 → 動作確認 → コミット」の小さいサイクルで進めること

これにより、セッションを跨いでも一貫した開発方針を保てました。

### Plan Mode

DBスキーマ（テーブル設計・RLSポリシー）、Next.jsのルーティング構成、実装サイクルの順序（どの機能から着手するか）をPlan Modeで設計し、内容を確認したうえで実装を開始しました。「いきなり書き始めない」ことで、後から設計をやり直す手戻りがありませんでした。

### Gitワークフロー

1機能実装したら、必ずブラウザで実際に操作して動作確認し、問題なければコミット・pushする、というサイクルを最後まで徹底しました。認証（F-01/F-02）→ 案件CRUD（F-04〜F-06）→ 応募状況一覧（F-07）→ 進捗・絞り込み（F-08/F-09）→ 納品・入金（F-10/F-11）→ 月次集計（F-12）→ ダッシュボード（F-03）という順で、小さい単位のコミットを積み重ねています。

### Claude in Chromeでの実機確認

実装のたびにブラウザを操作して画面を確認し、コンソールエラーの有無までチェックしながら進めました。この過程で、Next.js 16のAPI変更やUIライブラリの仕様差による不具合をその場で見つけて修正できています。

## 所要時間

要件定義からVercel公開・README作成まで、実装部分（Gitのコミット履歴ベース）で約3時間でした。要件定義書の作成にはこれとは別に事前の時間をかけています。

## つまずいたところ・解決方法

### Next.js 16で`middleware.ts`が`proxy.ts`に変わっていた

Next.js 16では、`middleware`という命名がExpress.jsのミドルウェアと混同されやすいという理由から、ファイル規約が`middleware.ts`から`proxy.ts`へ、関数名も`middleware`から`proxy`へ変更されていました。`node_modules/next/dist/docs`に同梱されているドキュメントを確認し、最新の規約に合わせて実装しました。

### shadcn/uiのBase UI移行

導入したshadcn/uiのテンプレートが、想定していたRadix UIではなくBase UIベースになっており、いくつか挙動が異なりました。

- `Button`をリンクとして描画する際は`asChild`ではなく`render`プロパティを使う必要があった
- `Select.Value`は選択中の値を自動でラベルに変換してくれず、`(value) => ラベル`という変換関数を明示的に渡す必要があった（渡さないと`interview`のような生の値がそのまま表示されてしまう）

いずれもブラウザでの実機確認中にコンソールの警告や表示崩れから気づき、その場で修正しました。

### Supabaseの新しいAPIキー体系

Supabaseのダッシュボード表記が、従来の「anon key」から「Publishable key」（`sb_publishable_...`）に変わっており、最初はどこに目的のキーがあるか分かりませんでした。

### メール確認フローの考慮漏れ

Supabaseプロジェクトはデフォルトでメールアドレス確認が有効なため、サインアップ直後はセッションが発行されません。当初はサインアップ後にそのままダッシュボードへ遷移させる実装にしていましたが、実際にAPIを叩いて検証したところセッションが返らないことが判明し、「確認メールを送信しました」という案内を表示する分岐を追加しました。
